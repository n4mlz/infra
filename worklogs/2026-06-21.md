## 判断

### kube-vip 移行

- addressless eth1 + Cilium L2 Announcement は Cilium kube-proxy replacement の要件と両立しないため、kube-vip に移行する
- kube-vip が leader node の eth1 に VIP を実際に付与して ARP で広報する方式に切り替える
- Cilium LB IPAM / L2 Announcement は完全に廃止する
- chart versions は kube-vip 0.9.1, kube-vip-cloud-provider 0.2.9 に固定

### Cilium Gateway API → Istio Gateway 移行

- Cilium Gateway API は TPROXY redirect が externally-facing NIC (eth1) に対して生成されず、外部 ingress 時に L7LB service backend が見つからない drop が発生する
- Cilium Gateway の eBPF+TPROXY 方式は CNI datapath に密結合しており、kube-vip で VIP を eth1 に付与しても L7 redirect chain が機能しない（cilium/cilium#43819 と類似の既知問題）
- Istio Gateway は通常の Deployment+Service として動作し、Cilium の TPROXY/L7LB 経路に依存しないためこの問題を回避できる
- 将来 Istio Ambient mesh を見据え、入口 Gateway も Istio に統一することで設計の連続性を確保する
- DSR/Geneve は試行したが L7LB (Gateway API) と非互換なため削除。Cilium は SNAT モードで十分
- IP pool は 163.220.236.73-76 の 4 個を維持
- kube-vip-cloud-provider に `allow-share-global: true` を設定し、将来 TCPRoute 等で複数 Service を同一 VIP に同居させられるようにする
