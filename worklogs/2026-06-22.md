## 実施内容

- `smoke-test.n4mlz.dev` の外部到達性を切り分けた
- DNS は `163.220.236.74` を返し、Gateway / HTTPRoute / Certificate / backend は cluster 内から正常に動作することを確認した
- 外部 TCP SYN は `wk-02 eth1` に到達していた
- Cilium DSR により Gateway Pod へ転送された後、SYN-ACK が `src=163.220.236.74` のまま eth0 へ出ていたことを確認した
- public VIP を source address とする reply traffic を eth1 に戻す policy route を一時適用し、HTTP/HTTPS が 200 OK になることを確認した
- DSR による client IP preservation は現時点の要件ではないため、SNAT mode も試した
- SNAT mode だけでは public VIP を source address とする reply が eth0 default route に従う問題は解消しないことを確認した
- public Gateway worker に node 用 public IP を持たせる案も試したが、`163.220.236.73-76` をすべて Service VIP に使う要件に反するため撤回した
- Cilium LoadBalancer mode を DSR/Geneve に戻し、public egress routing DaemonSet で Service VIP source traffic だけを eth1 に戻す構成にした
- Talos v1.9 の machine config で source-based `ip rule` を first-class に宣言する設定は見当たらなかったため、policy route は Kubernetes DaemonSet で管理する

## 判断

- 原因は Gateway/API/TLS ではなく、Cilium DSR reply の node egress route だった
- `externalTrafficPolicy: Local` だけでは reply が eth0 に出る問題は解消しないため、解決策にしない
- Talos machine config の通常 route は destination-based であり、addressless eth1 と Service VIP pool 節約を両立するには source-based routing が必要になる
- client IP preservation が不要でも、SNAT mode だけでは addressless eth1 の戻り経路問題は解決しない
- ただし SNAT/DSR のどちらでも、最終的に `src=<Service VIP>` の reply は node の main route に従う
- Service VIP pool を節約するため、worker eth1 に node 用 public IP は割り当てない
- 現時点の採用構成は DSR/Geneve + `public-egress-routing` DaemonSet
- DaemonSet manifest の先頭コメントと README に、Gateway Service VIP 範囲・public VLAN gateway・VLAN ID への依存を明記する
