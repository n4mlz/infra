- IaC に強く拘る。私 (n4mlz) の環境に specific な情報 (IPアドレスやホスト名など) は、出来る限り変数に移し、再利用しやすいようにする。また、変数だけ管理するファイルがあればそこだけ書き換えれば
- リポジトリを public にすることに留意する。つまり、secret な情報は全て 1password に退避する。また、.gitignore に頼りすぎたくない (ignore する目的がビルドの残骸などではなく secret であることなら、それは 1password に対比するべき)。
- 綺麗に、シンプルに書く。必要でない余計なことは書かない。リポジトリ内のコードは全て意味のある記述であるべき。
- デフォルトでその値であるものは、特に理由がなければ記述しない (自動生成されたファイルを除く)。コードは必要なもののみで構成されるべき。
- 自動生成されたファイルは、そのファイルが何からどのような手順によって生成されたかをそのディレクトリまたはその親ディレクトリの README.md 等に明記しておく (ファイルの再生成によって README.md が消失しないようにすること)。
- 作業ログは secret な情報を含まない範囲で全て worklogs/yyyy-mm-dd.md に記録する。
- このリポジトリ全体に関わる構造化した情報は docs に綺麗にまとめていく (ディレクトリは適宜作成して頂いて構わない)。
- 何か特定のものに specific な情報はそれに対応するディレクトリ配下に README.md を作成してまとめる。基本的には、手前のディレクトリほど大域的な情報を書き、ディレクトリがネストするごとにより粒度の細かい情報を書くようにする。
- ドキュメント用のファイルでなく通常のファイルである場合、ファイルの先頭にそのファイルの責務を doc コメントで簡潔に記述する。
- コード上に現れない暗黙的依存は、doc コメントに追記する。コード内のコメントは、英語で記述する。
- インクリメンタルに進める。そのため、まだ扱わないものに関してはリポジトリに含めない (意識する対象が増えて困惑するだけなので)。
- ドキュメントは日本語で記述する。
- 人の知識/記憶に依存しない構造にする。時間が経てば忘れるので、操作等は全てドキュメントに起こす。
- 不確実な情報がある場合は Web 検索を利用して調べ、確認する。ファクトチェックを行う。
- ワークアラウンドな解決をしない。そのような解決は解決ではない。
- レビューを受けた場合は、その指摘によって副次的に対応する必要のあるものを自発的に見て底まで含めて修正する。同様の指摘が該当する場所がないかを確認する。
- 勝手に commit を行わない。commit する前に、人間が必ず目を通し、承認された場合のみ commit する。git push 等リモートを変更する操作は必ず人間が行うので、必要がある場合は手順のみ提示する。**もし勝手な commit や push をした場合お前を解約する。そんな事をした場合以降再起不能にして二度とお前を使わないようにするので肝に銘じておくこと。**
